# 文档协作文风（Collaborative Documentation Writing Style）

本文档定义本仓库文档的**写作风格与审稿标准**，与「写哪篇、改哪块」的 scope 决策分离，专门约束「文章内部如何表达」。

---

## 一、服务对象与目标

### 这套文档服务谁

- **在 AI 协作开发场景下**，需要快速理解项目的人。
- 读者可能是新人、外部协作者，或隔一段时间再回来看代码的维护者。
- 目标是在有限时间内：**一眼看懂全局**，再按需深入，而不是先陷入细节。

### 这套文档不追求什么

- **不是**传统 API reference：不要求一开始就展开所有 API、参数、脚本细节。
- **不是**面向维护者的命令清单：脚本名、路径、产物列表不应成为文档主轴。
- **不是**技术细节穷举：不要求「把所有实现枝节写全」。
- **不是**维护者备忘录：读者不应需要先成为专家才能读懂。

### 什么叫「好的协作文档」

- 读者能**快速抓住**：大局流程、模块职责、设计用意、大致调用/数据关系。
- **概览优先于细节**：先建立心智模型，再给命令或清单。
- **流程优先于 API**：先讲「事情怎么发生」，再讲「每个部件叫什么」。
- **模块职责优先于实现枝节**：先讲「谁负责什么」，再讲实现细节。
- **设计动机优先于命令清单**：先讲「为什么这样设计」，再列怎么执行。
- 文档**支持跳读**：有小标题、有总述、有「继续深入」的入口，读者可先建立地图再按需深入。
- **图文协同**：图承担「结构、流程、关系」的传达，文承担解释与边界说明；不是纯文字堆砌。
- 有「**Dive Deeper**」入口，但**不在正文一开头就淹没读者**。

---

## 二、表达优先级与结构建议

写作时应遵循以下**表达顺序**（概览层 → 细节层）：

1. **一句话总述**：这篇文档在讲什么、解决什么问题。
2. **Why**：为什么需要这套能力/流程/设计；读者为何要关心。
3. **Big Picture**：整体图景——主要阶段、主要组件、数据/控制流关系（可用流程图或简短段落）。
4. **Main Flow**：从读者视角的主流程——「先做什么、再做什么、最后得到什么」。
5. **Module Roles**：各模块/脚本/Agent 的职责简述；按「谁负责什么决策、什么产出」组织，不按脚本名简单罗列。
6. **What You Usually Do**：平时最常用的入口、典型操作、决策树（例如：只想看影响 → A；要更新文档 → B；要验证 → C）；**前置条件（环境、CLI、依赖）应在此前或此处明确**。
7. **Boundaries / Pitfalls**：边界条件、易错点、阻塞 vs 非阻塞、不该做什么。
8. **Dive Deeper**：进一步阅读的链接（脚本 README、流程图文档、测试、源码等）。

**禁止或弱化**：

- 不要一上来就扔命令或脚本列表。
- 不要让脚本名、路径、产物列表成为文档主角。
- 不要用「产物列表」或「阶段列表」代替「叙事」——先讲清楚读者要解决什么问题、再给手段。

---

## 三、图文职责

- **图**（含 Mermaid 等）：承担「整体流程、阶段关系、模块依赖、数据流」的传达，帮助读者在脑中形成地图。
- **文**：承担动机解释、边界说明、前置条件、易错点、以及「接下来看哪里」。
- 图文应协同：有图时，正文不必重复图的全部信息，而是补充「为什么」「注意什么」「深入哪里」。

## 四、Diagram SOP

本节定义面向 `book/src` 等教学型文档的 **diagram writer / reviewer SOP**。目标不是“让 Mermaid 能渲染”，而是让图成为稳定的理解工具，而不是偶然好看的 AI 草图。

### 4.0 渲染、语义、视觉三类问题的分工

- **Render correctness** 属于 verify 层的 blocking 要求：Mermaid 语法错误、fence 错误、真实渲染失败，不应留给 reviewer 兜底。
- **Semantic quality** 属于 review / rework：图型不对、一图多义、该拆未拆 / 不该拆硬拆、图文边界不一致、细节没有下沉到后续章节。
- **Visual quality** 也属于 review / rework：拥挤、节点文案过长、线条挂大段解释、视觉重心不稳、overview 图不像正式文档地图。
- review 必须建立在**渲染后的图**上，而不只是 Mermaid 源码。render correctness 先通过 verify，再进入基于 screenshot 的 semantic / visual review。

### 4.1 先写 Diagram Spec，再写 Mermaid

- 凡是需要新增或实质修改 Mermaid 图，writer 必须先写一份 **diagram spec**，不能直接裸写 Mermaid。
- diagram spec 必须进入中间产物，建议按文档路径做 slug 目录，例如：`docs/generated/diagram_specs/book__src__01-overview/block-01.md`。
- spec 至少回答这些问题：
  - 图类型：`overview / sequence / boundary / coverage`
  - 这张图只回答哪个问题
  - 目标读者读完后应该获得什么心智模型
  - 这张图明确不展示什么
  - 哪些细节应该下沉到后续章节图
  - 预计主块 / 参与者有哪些
  - 为什么保留单图，或为什么要拆图
- reviewer 和 rework 必须把 spec 当成可引用证据：如果最终 Mermaid 偏离 spec 里的单一问题、边界或拆图判断，应明确指出“图已跑偏”。

### 4.2 什么时候需要图

- 当读者需要先建立**整体结构**、**主流程**、**时序关系**、**边界决策**或**风险覆盖关系**时，应优先考虑配图。
- 当正文已经在讲「谁先发生、谁依赖谁、哪里被阻止、哪类风险由哪组测试兜底」时，图通常能显著降低理解成本。
- 当没有图就容易退化成长段说明、模块清单或口头流程图时，应补图。

### 4.3 什么时候不该硬画图

- 当内容只是一个局部事实、单个命令、短列表或一段简单定义时，不要为了“有图”而画图。
- 当图只会重复正文、不能新增理解收益时，不要硬加 Mermaid。
- 当一张图必须同时承担两种以上主语义，且拆图更清楚时，应该拆图，不要强行塞进一张图。

### 4.4 图型选择

- **Architecture / Overview 图**：用于解释系统整体结构、主组件关系、阶段分层。  
  适合：overview、系统概览、模块关系。  
  不要混入：详细时序、过多失败分支、测试文件清单、过深实现细节。
- **Sequence / Streaming 图**：用于解释事件顺序、消息流、增量组装、请求与回调链。  
  适合：流式解析、交互回合、请求-响应时序。  
  不要混入：模块分类、权限矩阵、覆盖关系。
- **Boundary / Approval 图**：用于解释允许/阻止、注册/审批/执行、停机/继续等闸门关系。  
  适合：安全边界、策略门、执行约束。  
  不要混入：完整系统概览、测试套件映射。
- **Coverage / Testing 图**：用于解释风险面、行为合同、代表测试簇之间的覆盖关系。  
  适合：测试策略、风险覆盖、可信性说明。  
  不要混入：完整运行时主流程或模块内部时序。

### 4.5 Mermaid 与视觉表达规则

- **每图只讲一件事**：先写一句“这张图的主语义是什么”，若一句话说不清，通常说明图义混杂。
- **清晰度优先于节点数量**：主节点数量不是硬约束。只要图仍能保持单一语义、稳定阅读路径、低交叉连线、明确分组和短而一致的命名，就允许较多节点存在。
- **当清晰度无法维持时必须拆图**：不要为了把信息塞进一张图而继续压缩字号、堆叠标签或补更多解释文字。拆图是成熟表达，不是失败。
- **分组优先于连线**：先明确阶段、层级、闸门或测试簇，再决定箭头；不要用箭头替代分组。
- **默认单方向布局**：优先 `LR` 或 `TD` 的单主方向；除时序图外，尽量避免来回折返。
- **尽量避免交叉箭头**：一旦交叉明显增多，优先拆图或重分组，不靠“再多加几条线”补救。
- **命名短而一致**：节点名优先写职责或阶段，不优先写内部类名、变量名、测试文件名。
- **Overview 图先像地图，再像解释图**：big picture 图优先呈现块与关系，不优先展开实现细节。
- **节点标题要短、名词化、可扫读**：不要把总结句、条件句或解释性短段直接塞进节点。
- **解释句尽量进 caption，不挂在线上**：连线文字只保留最必要的关系词，长解释应放到图后说明。
- **主视觉重心要稳**：避免把噪音节点、零散容器或大段说明文字堆在画面中心。
- **容器与分组要克制**：只有当分组真的在表达阶段、层级或职责边界时才画容器；不要把每个节点都再包一层。
- **细节必须下沉**：overview 图承担地图职责；实现链路、异常分支、tool-call 组装、审批细则应下沉到后续章节专属图。
- **图成立时不要机械拆图**：如果单图已经清晰、单义、阅读路径稳定，就不要为了“显得做了事”而硬拆。
- **不要依赖颜色传义**：Mermaid 在不同环境主题下颜色不稳定，语义必须靠结构、分组、标题和文字表达。
- **Caption 必须说明边界**：每张图后必须有简短 caption，至少回答：
  - 这张图展示什么
  - 这张图不展示什么
  - 读者应如何阅读它

### 4.6 何时必须拆图

当下列条件中满足任意两条及以上时，应至少判定为**建议拆图**；若已经影响主语义判断或正文需要反复补救，应判定为**必须拆图**：

- 同一张图同时表达结构关系和时间顺序。
- 同一张图同时表达主流程和大量边界 / 异常说明。
- 读图顺序无法自然保持左到右或上到下。
- 为了讲清楚不得不使用很多长标签。
- 交叉箭头明显增多。
- caption 已经需要解释“这张图其实分几部分看”。
- 图中存在多个并列主问题，而不是单一图意。

### 4.7 推荐拆图策略

- **Overview / Architecture 图过重时**：
  - 图 1：系统结构图，只讲模块、分组与主依赖。
  - 图 2：主任务流图，只讲任务如何穿过系统。
- **Streaming 图过重时**：
  - 图 1：传输与解析链，只讲 HTTP / SSE / event 解释。
  - 图 2：tool-call 组装链，只讲碎片如何变成完整调用。
- **Tools / Safety 图过重时**：
  - 图 1：控制链，只讲注册 / 审批 / 执行 / 停机。
  - 图 2：风险分层，只讲不同工具边界强弱与误解。
- **Testing 图过重时**：
  - 图 1：风险面地图，只讲系统最怕什么失控。
  - 图 2：测试簇与行为合同映射，只讲哪些测试在证明哪些边界。

### 4.8 常见反模式

- **一图多义**：同一张图同时想讲架构、时序、边界和测试覆盖。
- **模块、时序、边界混画**：读者不知道箭头表示调用、阶段推进，还是审批关系。
- **箭头来回穿插**：需要靠追线才能理解主流程。
- **节点命名过长**：节点本身像一句说明文，遮蔽图的结构。
- **线条上挂大段解释文字**：图面先被文案占满，关系反而退居次要。
- **画面中心堆噪音块**：把边角说明、实现备注或例外分支压在视觉中心，破坏主图重心。
- **容器过多、分组太碎**：读者先看到盒子套盒子，而不是主要结构。
- **过度依赖颜色表达信息**：换主题后语义消失。
- **把实现细节塞进 overview 图**：让系统总览图退化成源码索引。
- **把 overview 图写成“实现混合图”**：读者本应先拿地图，却被迫先消化 `LLM bridge`、`assembler`、工具分类等下沉细节。
- **把 testing 图画成测试文件清单**：读者只看到文件名，不知道这些测试在证明什么。
- **明明该拆图却硬压成一张**：最后只能靠超长 caption 或正文补丁解释图到底怎么看。
- **明明单图成立却被机械拆图**：把原本顺滑的一条主线切碎，读者反而要来回对照多张图。

### 4.9 Diagram Reviewer SOP

#### Must Fix

- Mermaid fence 或渲染本身已经失败。此类问题应先由 verify 阻塞，不应等 reviewer 兜底。
- 图与正文主语义冲突，读者会因此误解系统、流程或边界。
- 图型选错，导致读者把分类关系误当流程、把覆盖关系误当时序。
- 一图多义严重，读者无法判断箭头含义。
- 图文不一致：正文说 A，图画成 B。
- 图完全不能帮助理解，甚至比没有图更误导。
- 已出现“建议拆图”条件中的多项，且主语义已经依赖正文额外解释才能成立。
- overview 图不再像地图，而像实现混合图；关键细节没有下沉到后续章节图。
- 基于 screenshot 可见明显视觉失真：主视觉重心紊乱、中心噪音块压住主线、长句节点或长句连线已破坏扫读。

#### Should Fix

- 节点增加后，清晰度开始下降，但还未严重到必须拆图。
- 分组弱，视觉秩序差。
- 线条交叉过多。
- 节点命名不一致或过度内部实现化。
- 图只是重复正文，没有新增理解增量。
- 缺少 caption，或 caption 不能解释图的边界。
- 已满足“建议拆图”条件中的两项以上，但仍可通过拆图或重排快速修复。
- 基于 screenshot 可见轻中度拥挤、容器过多、文案偏长，但仍可通过重排或缩短标签修复。

#### Reviewer 输出要求

- 必须先区分问题类别：**render correctness / semantic quality / visual quality**。
- diagram review 必须读取 diagram spec 和渲染 screenshot；不要只看 Mermaid fence 文本。
- 必须显式回答这 5 个问题，而不是泛泛说“图不够清楚”：
  - 这张图是不是只回答一个主问题？
  - 细节有没有下沉到后续章节？
  - 标题是不是短而可扫读？
  - 画面中心有没有噪音？
  - caption 有没有承担解释职责？
- 必须指出问题属于哪类：**图型错误 / 语义混用 / 阅读路径不稳 / 交叉过多 / 分组弱 / 图文不一致 / caption 缺失或失效 / 应拆图未拆**。
- 每个 diagram 问题都必须进一步标注为 **semantic quality** 或 **visual quality**，避免把“图型错误”和“画面拥挤”混成同一种反馈。
- 必须指出该图应该保留的主语义，而不是笼统说“画得不够好”。
- 建议必须可返工：明确是应该**拆图**、**改图型**、**重排分组**、**缩短命名**，还是**补 caption**。
- 当判断需要拆图时，必须指出推荐拆法：例如“拆成结构图 + 主流程图”或“拆成传输解析链 + tool-call 组装链”，而不是只写“图太复杂”。

## 五、审稿判断标准（Clarity Reviewer）

审稿时，除事实正确性与格式外，应按以下标准检查**人类理解效果**：

- **是否先给大局**：读者能否在读完前半部分后，快速理解系统/流程的目标与整体形状？
- **是否先给地图再给细节**：是否先有总述、Why、Big Picture/Main Flow，再出现命令、路径、产物列表？
- **是否命令先于理解**：是否在读者还不知道「这是在干什么」之前就出现大段命令或脚本名？若是，应标记为问题。
- **是否清单过多、叙事过少**：是否读起来像维护者备忘录或脚本清单，而不是「帮人看懂」的叙述？若是，应建议加强动机与流程叙述。
- **模块职责是否讲清楚**：读者是否能回答「谁负责什么决策、什么产出」？
- **是否解释了设计用意**：关键边界（例如为何只改 README + book、为何由 Agent 做 scope 决策）是否有简短解释？
- **是否给出深入阅读入口**：文末或相关小节是否有明确的「进一步阅读」链接？
- **是否存在「信息对但无助于理解」**：某段是否事实正确但既不建立地图也不解释动机？若是，应建议改写为「先建立心智模型再给细节」。

审稿**不应**退化为：只检查格式、错别字、链接是否有效；这些由验证脚本和 fact-check 覆盖。Clarity 审稿专注「读者能不能快速理解项目」。

## 六、推荐骨架（按文档类型）

以下为**推荐骨架**，不是强制模板；可根据具体文档微调，但建议保持「概览优先、流程优先、支持跳读」的心智框架。

### README 型

- 一句话总述（本段/本页在讲什么）。
- 为何重要 / 适用场景（Why）。
- 整体图（主要组件或流程，可配图）。
- 常用操作（What you usually do）：决策引导 + 1～3 个典型命令/入口，并注明前置条件。
- 边界 / 注意（Boundaries, pitfalls）。
- 深入阅读（Dive deeper：链接到 book 章节、脚本 README、流程图文档等）。

### Book 章节型

- 一句话总述 + 本章在全书中的位置（可选）。
- Why：为什么需要这一块能力/流程。
- Big Picture：整体流程或架构（可 Mermaid 或简短段落）。
- Main Flow：读者视角的主流程。
- Module Roles：各部件职责简述。
- What You Usually Do：常用入口与前置条件。
- Boundaries / Pitfalls。
- Dive Deeper：相关脚本、文档、测试。

### 流程说明型

- 目的（Why this flow exists）。
- 主流程图 / 阶段一览（Big Picture）。
- 各阶段职责（Module Roles）。
- 常用入口（What you usually do）与前置条件。
- 产物与校验边界（Outputs & verification）。
- 深入阅读。

### 系统概览型

- 系统目标（一句话 + Why）。
- 架构 / 模块关系（Big Picture）。
- 主流程（Main Flow）。

## 七、与现有流程的关系

- **Scope decision、change_facts、reference_context、verify** 解决的是「写哪篇文档、改哪块范围、用什么事实」。
- **本文档 + docgen-overview-writer + doc-clarity-reviewer** 解决的是「文章内部怎么写、审稿时如何判断理解效果」。
- 写作任务（如 `ai_doc_restructure_or_update`）应同时读取本文档与 overview-writer skill；审稿任务（如 `ai_review`）应同时读取本文档的审稿标准与 doc-clarity-reviewer skill。
- 当文档包含 Mermaid 图时，diagram 写作与审稿同样遵循本文档：写作端先产出 diagram spec，再按 Diagram Writer SOP 生成 Mermaid；verify 端产出 render + screenshot artifacts；审稿端按 Diagram Reviewer SOP 基于 spec 与 screenshot 判断是否可读、是否与正文一致、是否需要返工。
- 规范与 skill 的引用方式见 [AGENTS.md](../AGENTS.md) 的 Documentation Automation Rules 与各 task 文件的 Step 0。
